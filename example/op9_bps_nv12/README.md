# OnePlus 9 — Native NV12 from the Spectra/BPS ISP into openpilot camerad

Reverse-engineering guide for driving the OnePlus 9 (SM8350 / Snapdragon 888,
Spectra **Titan-480** ISP) **BPS** (Bayer Processing Segment, the offline ISP on
the ICP/A5 firmware) to demosaic the IFE RAW stream into **NV12** and feed it
directly into openpilot's `camerad` — replacing the RAW output path.

> Status as of 2026-06-26. The whole pipeline is wired, builds, runs to firmware
> execution, and the BPS config is **validated against the stock HAL**. One wall
> remains: the ICP firmware aborts with `FAR=0x9101e0`. This has been **proven
> on-device to be camerad's per-frame BUFFER setup, not the blobs** — the
> stock-exact blobs reproduce the identical abort. The root-cause derivation of
> `0x9101e0` is the open item.

---

## 1. Goal & current status

**Goal:** `IFE RAW (Bayer) → BPS demosaic → NV12` written to camerad's VisionIPC
buffer, so openpilot's WIDE_ROAD camera (imx689, 4000×3000) gets a processed
image without a CPU/GPU demosaic.

**Why BPS and not the IFE:** an exhaustive prior effort proved the single-IFE
pixel pipe **cannot** emit native 4:2:0 NV12 on this part — the FULL_DISP luma is
an odd height (4000×2999, demosaic-cropped) that the line-based write-master
rejects, and luma/chroma share a config block so you get even-luma XOR
4:2:0-chroma, never both. The stock HAL finishes NV12 in the **IPE/ICP** (post-IFE),
not the IFE. So the supported route is the offline **BPS** (RAW→YUV), which the
stock HAL uses for stills/reprocess at *exactly* camerad's 4000×3000 geometry.

**What works today:**
- ICP firmware downloads (`CICP.FW.1.0-00099`, Titan-480, API_V2).
- BPS **acquire** succeeds with the captured `bps_cfg` io_config.
- `config_bps` CLK V2 blob accepted.
- **IFE RAW frames flow** into BPS (`Buf done VFE:4 res_id 0x3006`, addrs advancing).
- The firmware **executes the BPS job**, then aborts.

**UPDATE (2026-06-26): the `FAR=0x9101e0` abort is SOLVED — it was the `bps_cmd`
(`CAM_CMD_BUF_FW`) buffer, not the image blobs.** An iova-layout experiment
(allocate the dummy first so it blankets the low icp iova) proved the FAR was
blob-relative: it moved `0x9101e0 → 0x1cc01e0`, always `base + ALIGN(464,32)` =
`bps_cmd`'s per-slot stride. Two fixes in `spectra.cc` `configICP`/`config_bps`:
1. `bps_cmd.init(..., /*cached=*/false, icp_device_iommu)` — **UNCACHED**, single
   contiguous buffer (was cached + an `ife_buf_depth` ring). The FW's cache-INVALIDATE
   (DCIMVAC) of the cached FW cmd buffer faulted because `bps_cmd` isn't in the FW's
   cache-maintenance MMU context; uncached skips it. **This is the key fix.**
2. `buf_desc[0].offset = 0` (single in-flight frame slot) — was `aligned_size()*idx`,
   whose non-zero slots weren't mapped.

⇒ **The firmware now executes the full BPS image pipeline.** The FAR jumped off
the cmd buffer entirely to `FAR=0xdfd00000` — a *new* fault in the data-buffer
range (a stable, computed write-buffer-invalidate address past all input/output
buffers). That is the current frontier; see §6.

**Original wall (now diagnosed):** `Data Abort at 0x302e0, FSR=0x807, FAR=0x9101e0`
→ A5 watchdog → ICP recovery → camerad `config_ife` assert. Reproduced with the
stock-exact config, which is what led to isolating it to `bps_cmd`; see §6.

## 2. Architecture

### Pipeline
```
imx689 sensor ──CSID/RDI──> IFE (raw passthrough) ──RAW Bayer 4000x3000 PLAIN16_10 stride 8000──>
   BPS (offline ISP, runs on the A5/ICP firmware) ──demosaic + IQ──> NV12 4000x3000 ──> camerad VisionBuf
```

camerad's `ISP_BPS_PROCESSED` output type drives `configICP()` (one-time ICP/BPS
acquire) + `config_bps()` (per-request job submission). The road camera output
type lives in `hw.h` (WIDE_ROAD → `ISP_BPS_PROCESSED`, gated by `OP9_BPS` env).

### The three blobs camerad supplies
The BPS job needs three opaque blobs that the stock CamX HAL normally generates
at runtime. We **capture them from the stock HAL** (§4) and bake them into
`bps_blobs.h`:

| blob | what it is | size (imx689 multi-out) |
|------|------------|-------------------------|
| `bps_cfg` | BPS acquire **io_config** (port geometry: in + OUT_FULL + DS4/16/64) | 1248 B |
| `bps_settings` | **BPSIQSettings** (per-module IQ params) | 4096 B |
| `bps_striping_output` | striping-library output (per-stripe pixel geometry/DMA descriptors) | 21736 B |

The **CDM** (the register program that drives the BPS hardware) is **generated by
the firmware** from `bps_cfg + striping + settings`; camerad only allocates the
CDM buffer (size `0x214e0`, validated). So we supply 3 blobs, not 4.

### `bps_frame_process_data` ABI (the per-frame command buffer, "cb0")
The OP9 SM8350 firmware reads `struct bps_frame_process_data`
(`hfi_session_defs.h:312`). The original comma code filled the wrong (tici /
Spectra-280) layout; the correct OP9 layout, as words in the dumped cb0:

```
word  0  max_num_cores            (0)
word  1  target_time              (0x09c9ef70)
word  2  ubwc_stats_buffer_addr   (0)
word  3  ubwc_stats_buffer_size   (0)
word  4  cdm_buffer_addr
word  5  cdm_buffer_size          (0x214e0)
word  6  iq_settings_addr
word  7  strip_lib_out_addr
word  8  cdm_prog_addr
word  9  request_id
word 10+ frame_buffer buffers[9]  @ byte 0x28
```
Each `frame_buffer` = `{ uint32_t buf_ptr[2]; uint32_t meta_buf_ptr[2]; }`
(16 B, `MAX_NUM_OF_IMAGE_PLANES=2`). Patch offsets within a buffer:
`buf_ptr[0] = 0x28 + b*16`, `meta_buf_ptr[1] = 0x34 + b*16`.

**BPS port enum** (the `buffers[]` index):
```
0 = INPUT (RAW)        5 = STATS BG
1 = OUTPUT_FULL (NV12) 6 = STATS BHIST
2 = DS4                7 = REG1
3 = DS16               8 = REG2
4 = DS64
```
The stock HAL puts the **input** in `buffers[0].meta_buf_ptr[1]` (with
`meta_buf_ptr[0]=1` as a flag) and the **NV12 output** in
`buffers[1].meta_buf_ptr[1]` (Y base only; the firmware derives UV = Y+uv_offset).

## 3. The standalone striping-library harness

The striping output (`bps_striping_output`) is produced by
`/vendor/lib64/libipebpsstriping.so`. Running it standalone lets you regenerate
the blob for any config and — critically — read the **real buffer sizes** the
firmware expects.

### Namespace-escape technique (the hard part)
`libipebpsstriping.so`'s `DT_NEEDED` pulls in the whole vendor graphics/hidl/
gralloc chain (→ `libutils` → `libunwindstack` → `Xzs_Construct` unresolved), so a
plain `dlopen` outside `cameraserver`'s linker namespace fails. But:

```
$ aarch64-linux-gnu-nm -D --undefined-only libipebpsstriping.so
```
shows its **only imported symbols are 19 `@LIBC` ones** (fopen/malloc/memcpy/
abort/`__cfi_slowpath`/`__cxa_finalize`/`__stack_chk_fail`/…). All the heavy
`DT_NEEDED` libs are over-linked and **unused**.

**Fix:** build **30 empty stub `.so`** (one dummy symbol each) with the matching
SONAMEs (`libc++.so`, `libcutils.so`, `liblog.so`, `libutils.so`, `libhidlbase.so`,
`libgralloc.qti.so`, all `android.hardware.graphics.*@X.Y` + `vendor.qti.hardware.
display.*` HALs, …). Put them **first** in `LD_LIBRARY_PATH` → `dlopen` loads the
empty stubs, cutting the transitive chain; the lib loads with only the real libc.

```bash
# stubs (host, AOSP clang). dummy.c MUST contain a symbol, e.g. `static int x=0;`
clang-r416183b/bin/clang --target=aarch64-linux-android21 -shared -nostdlib \
    -fuse-ld=lld -Wl,-soname,<NAME> -o stubs/<NAME> dummy.c
# iterate the SONAME list in BASH (zsh won't word-split $LIBS)
```
Build the harness itself with **termux clang** on-device (termux is bionic, so it
emits a normal Android exe; bionic dedups libc by SONAME so the dlopen'd lib
reuses one libc):
```bash
/data/data/com.termux/files/usr/bin/clang gen2.c -ldl -o gen2
LD_LIBRARY_PATH=/data/local/tmp/stubs:$PREFIX/lib:/system/lib64 \
    ./gen2 cfg.bin settings.bin /vendor/lib64/libipebpsstriping.so out.bin   # under su
```

### Validated ABI
Exports are CFI thunks at `0x639xx`; the real bodies are the first `b` target.
```c
// real @ 0x2a958
int32_t BPSStripingLibraryContextCreate(
    const void* cfg, void* mem /*=NULL*/, uint32_t titanVer /*=0x60400*/,
    uint32_t hwVer /*=0x60400*/, void** handle, uint32_t* bufSize, uint32_t* cdmSize);
// shared BPS/IPE
int32_t IPEStripingLibraryCalculateScratchBufSize(const void* handle, uint32_t* sz);
// real @ 0x2b690
int32_t BPSStripingLibraryExecute(void* handle, const void* exe,
    void* out, void* meta, void* scratch);
```
- Context magic: `*(uint64_t*)handle == 0x425053534c435458` ("XTCLSSPB").
- `mem = NULL` is **correct** — Create `cbz x1` at `0x2aadc`: NULL → uses the lib's
  own default pointers (`adrp x0,0x63980 / x10,0x63988`); non-NULL → `ldp x0,x10,[x1]`
  (so `mem={ptr,ptr}` *replaces* those defaults; passing raw buffers = garbage-as-ptr,
  crashes Create).
- `exe` struct = `{ const void* iq @0; uint64_t @8 (value, not ptr, 0 ok); uint32_t
  cores @16 }` — **not** `{iq, cores@8}`.
- `CalculateScratchBufSize` → **41185728** (39 MB scratch that Execute needs).

### Validation & the prize (real sizes)
Create's `cdmSize` for the multi-output cfg = **`0x214e0` (136416)** which
**exactly matches the device-captured `cdm_size`** from the firmware dump — proof
that Create reproduces `cameraserver` and that the cfg + titanVer are correct.

| config | bufSize (striping) | cdmSize |
|--------|--------------------|---------|
| multi-output (in + OUT_FULL + DS4/16/64) | **21736** (0x54e8) | **136416** (0x214e0) |
| single-output (in + OUT_FULL, DS zeroed) | 14248 | 77216 (0x12da0) |

### UNSOLVED: `BPSStripingLibraryExecute` SIGSEGV
Execute crashes (fault addr `0x6` = NULL+6) at `lib+0x2c814`
`ldur x8,[x1,#6]` where `x1 = *(x4+0x3cc) = NULL`. `x4` is a **lib-internal malloc**
(a node-list element copied `x4→x5` in the `0x2c600` struct-copy block), independent
of iq content (zeroed iq same), cfg port count, buffer over-alloc, meta, and cores.

Two conclusive static-RE findings:
1. `mem=NULL` is correct (above) — not the cause.
2. **Offset `+0x3cc` is READ but NEVER WRITTEN anywhere in the lib**
   (`objdump -d | grep '#0x3cc'` = exactly 2 hits: the crash read `0x2c7e8` and an
   unrelated fprintf format-string adrp `0x4ac78`). So `+0x3cc` is an **external
   pointer the caller (CamX) pre-sets in the node template**, or a struct-version
   subtlety where Execute reads past a shorter node.

⇒ Cracking Execute needs the **CamX call context**, not more static RE: either
(a) **hook the live `cameraserver`** (LD_PRELOAD / ptrace a shim over
`BPSStripingLibraryExecute`) to capture the real 5 args + the node templates, or
(b) find CamX `camxbpsstriping.cpp` / `camxstripinglibincludes` for the
`BPSStripingLibExecuteParams` + node struct.

**Fallback (used today):** the captured multi-output blobs are already correct
(cdmSize validated) — Execute is not strictly required; just size the striping at
the real 21736 B and use the multi-output cfg with dummy DS buffers.

## 4. Captured blobs

Captured from the stock HAL by driving a **real photo via Open Camera**
(`net.sourceforge.opencamera`, `input keyevent 27`), which completes
RAW→BPS→JPEG at 4000×3000 (= camerad's imx689 geometry). The kernel dump module
(`op9_bps_dump=1`) logs the BPS acquire IO_CONFIG + per-frame FW frame data +
cb hex + a patch-walk; `kmsgcat` captures it losslessly.

- **`bps_cfg`** = 1248 B, the acquire IO_CONFIG. Layout = 9 × `ImageDescriptor`[96]
  + `secureMode`. Ports: `img[0]` INPUT BAYER fmt6 4000×3000 Ystride 8000; `img[1]`
  OUT_FULL NV12 fmt20 4000×3000 (Y/UV stride @0xa0/0xa8 — edit 4000→4096 for
  camerad's `MSM_MEDIA_ALIGN(4000,128)=4096` buffer); `img[2..4]` DS4 1000×750 /
  DS16 250×188 / DS64 64×48 fmt10; `img[5..8]` zero.
- **`bps_settings`** = 4096 B (BPSIQSettings).
- **`bps_striping_output`** = **21736 B**. The real meaningful data is the first
  16384 B; the rest is zero pad to the buffer size. Header `u16` = `14, 21, 2, …,
  2999` (= H-1). This was historically truncated (3160 then 16384); the correct
  size came from the harness Create `bufSize`.

Generator: `gen_bps_blobs.py` (host) emits `bps_blobs.h` as
`unsigned char bps_<x>[6][LEN]` indexed by `ImageSensor` enum; **imx689 = slot 5**
(`0 unk, 1 ar0231, 2 ox03c10, 3 os04c10, 4 imx766, 5 imx689`), others zero.

## 5. camerad integration (`spectra.cc`)

`config_bps()` builds the per-frame packet. Key points:

- **Frame-buffer patches** (`add_patch`): input → `buffers[0].meta_buf_ptr[1]`;
  output → `buffers[1].meta_buf_ptr[1]` (Y base). Match the stock: leave
  `buf_ptr[0]/buf_ptr[1]=0` and let the firmware derive UV. (Setting
  `buffers[1].buf_ptr[1]=UV` was tried — did not change the abort.) Ports
  `buffers[2..8]` are patched to a `bps_fullres_dummy` so the FW never writes at
  base 0.
- **Scalar header**: `iq_settings_addr → bps_iq.handle`,
  `strip_lib_out_addr → bps_striping.handle`,
  `cdm_buffer_addr → bps_cdm_striping_bl.handle` (allocated `0x214e0`),
  `cdm_prog_addr → bps_cdm_program_array.handle`.
- **CLK blob must be V2** — the BPS context `bw_config_version` is V2; comma's V1
  (`type=0x1`) is rejected ("Mismatch blob versions 2 expected v1"). Replay the
  captured 140 B V2 blob (`blob_type 0x5 = GENERIC_BLOB_CLK_V2`) verbatim.
- **Ordering**: `configICP()` must run **before** `camera_map_bufs()` (else
  `icp_dev_handle=0` at map time → RAW buf not mapped to the ICP iommu →
  `get_iova failed for patch`).
- **Hand-built `cdm_prog`** (`config_bps` ~L692-721): camerad constructs the BPS
  IQ register program with `write_cont`/`write_dmi`:
  `0x2868` white balance, `0x2878` debayer, `0x1868-0x1898` linearization,
  `0x2e68` CCM, plus `write_dmi` LUTs `0x1808` (linearization) / `0x3208` (gamma).
  The LUTs are patched at L836-839 to `bps_linearization_lut` / `bps_gamma_lut`,
  which are allocated in `configICP` (L1673/1694) on `m->icp_device_iommu`.
  camerad uses `sensor->linearization_lut` / `gamma_lut_rgb` sizes (= the IFE LUT
  sizes — the BPS block may expect different; a suspect, see §8).

The fully-wired source is `spectra_device.cc.abi_buffers` (this repo's
`spectra.cc`).

## 6. The firmware crash diagnosis

### The fault, decoded from the firmware
`CAMERA_ICP.elf` is a readable **ELF32 ARM** (the inner ELF starts 32 B into the
MBN-wrapped `/vendor/firmware/CAMERA_ICP.elf`). `aarch64-linux-gnu-objdump -d`
works.

- `PC=0x302e0` = `mcr p15,0,r0,cr7,cr6,{1}` (**DCIMVAC**, data-cache
  invalidate-by-MVA) inside `mmu_dcache_inval_region` (`0x302bc`), which loops the
  cache op over `[r0, r1)`.
- Its only caller is `vSYS_MaintainCache` (`0x7064`, Thumb). `desc r0 =
  {type@0, op@4, start@8, size@12}`. For `type==1` it does `ldrd start,size=[desc+8]`
  then `mmu_dcache_inval_region(start, start+size)`. (`type==0` = ICIALLU.)
- **9 call sites** to `vSYS_MaintainCache`: `0x719e, 0x88fa, 0x890e, 0x8a9c,
  0x8ab0, 0x9114, 0xb52c, 0xcd1c, 0x12a7a`.
- `FAR=0x9101e0` = the first unmapped address the cache walk hits.

So the FW is cache-maintaining a buffer whose extent reaches `0x9101e0`, which is
unmapped in camerad's ICP-iova space.

### Proven on-device: it's the BUFFERS, not the blobs
Built `bps_blobs_multi.h` = multi-output cfg (`imx689_bps_cfg_multi4096.bin`,
OUT_FULL stride→4096) + 21736 striping + settings — i.e. the **exact
stock-validated config** — and camerad **still aborts at the identical
`FAR=0x9101e0`**. `0x9101e0` is **invariant** across:
single ↔ multi cfg, 16384 ↔ 21736 striping, with/without `buffers[1].buf_ptr[1]`.

### Stock vs camerad frame buffer
The config command buffer **cb1 is byte-identical** stock vs camerad — the only
difference is cb0's buffer addresses. Decode:

| | stock (works, `bps_photo2.txt`) | camerad (aborts) |
|---|---|---|
| INPUT | `buffers[0].meta[1]=raw`, `meta[0]=1` | same (raw @0xd9a00000) |
| OUT_FULL | `buffers[1].meta[1]=Y` only (buf_ptr=0) | `meta[1]=Y` (Y/UV @0xd8600000/0xd91c0000, both mapped) |
| DS4 | `buffers[2].buf_ptr[0]+meta[1]` real bufs | dummy (0x980000, 20 MB) |
| blob iovas | strip/cdm/iq ≈ **0x13c0000** (19 MB) | iq/cdm_prog/strip/cdm = **0x920000/0x930000/0x940000/0x950000** (9 MB, 64 KB-spaced), dummy 0x980000 |

`0x9101e0` = `0x910000 + 0x1e0` — **one 64 KB slot below the lowest blob**
(iq@0x920000); nothing is mapped there. It is **not present in any blob, the cfg,
the iq settings, the striping, or the firmware ELF** (all grep'd) — the firmware
**computes** it. It is **non-page-aligned**, and page `0x910000` is unmapped, so it
is a computed sub-buffer base `≤0x9101e0` that the FW cache-maintains. Notably the
camerad ICP buffers land at ~9 MB iovas while the stock's were at ~19 MB — the FW
computes a write/scratch address that is mapped in the stock's layout but unmapped
in camerad's.

(Earlier ruled out by separate tests: output IOVA in all 4 frame_buffer fields →
same FAR; `cdm_prog=0` → same; single-output cfg → same. The output NV12 buffer
itself is correctly placed and mapped.)

## 7. On-device test loop (no reboot)

The device is **fully testable without a reboot**: the patched `camera.ko` is the
loaded `camera` module (verify: `/sys/module/camera/parameters/op9_bps_dump`,
`op9_tolerate_violation`, `op9_tolerate_pp` exist), 22 subdevs healthy.

`run_bps_test.sh` → `op9_bps_iter.sh` does, in ~2-3 min/iter:
1. deploy `bps_blobs.h` + `spectra.cc` into the proot openpilot tree,
2. `scons -j6 system/camerad/camerad` (proot),
3. set knobs (`op9_bps_dump=1`, `op9_tolerate_violation=1`, `op9_tolerate_pp=1`),
4. `setprop ctl.stop vendor.camera-provider-2-4`,
5. start `kmsgcat` in the background,
6. run camerad 14 s: `OP9_BPS=1 OP9_SINGLE_IFE=1 OP9_RDI=1 DISABLE_ROAD=1
   DISABLE_DRIVER=1 ./system/camerad/camerad`,
7. grep the kmsg for `FAR=` and the `[op9bps]` cb0/PATCH dumps,
8. `setprop ctl.start vendor.camera-provider-2-4`.

**Gotchas:**
- A crash-interrupted scons corrupts `.sconsign.dblite` → next build silently
  no-ops. Fix: `rm -f .sconsign.dblite system/camerad/**/*.o
  system/camerad/camerad` and confirm "scons: done building targets".
- Don't `rmmod camera` with the provider live (refcount>0) — panic. After a reboot
  the device FBE-relocks and termux sshd (CE storage) is down until the user
  enters the PIN.
- Long streams with unresolved CCIF/violations eventually destabilize the camera
  subsystem → reboot to recover.

## 8. Reproduce & next steps

### Connect
```bash
ssh -i ~/.ssh/oneplus_termux -p 8022 root@oneplus     # termux sshd, user u0_a186
su                                                     # magisk root
# proot (glibc ubuntu w/ the openpilot tree):
export PREFIX=/data/data/com.termux/files/usr
export PATH=$PREFIX/bin:$PREFIX/bin/applets:/system/bin
proot-distro login ubuntu -- /bin/bash -lc 'cd /root/openpilot && ...'
```
openpilot tree:
`$PREFIX/var/lib/proot-distro/containers/ubuntu/rootfs/root/openpilot`.

### Two concrete leads for `0x9101e0`
1. **Deep firmware trace (definitive):** disassemble the BPS job-processing path
   and find which of the 9 `vSYS_MaintainCache` call sites sets `desc.start =
   0x9101e0`, and what base + offset produces it (the BPS frame buffer-descriptor /
   an internal scratch table). That pins the exact buffer camerad is
   mis-providing/mis-sizing. `aarch64-linux-gnu-objdump -d CAMERA_ICP.elf`.
2. **iova-layout experiment (cheap, on-device):** the FW computes an address below
   camerad's ICP allocation base (~0x920000) where the stock's base (~0x13c0000)
   would be mapped. Force camerad's `configICP` ICP allocations to a higher base /
   the stock's order and re-run; if the FAR moves, it's iova-layout-dependent and
   the fix is to reserve/align that region.

### Other untested suspects
- camerad's **hand-built `cdm_prog`** (§5) may program a Titan-480 register
  wrongly; capture the stock `cdm_prog` (FW dump `cdm_prog_addr`) and diff.
- BPS **linearization/gamma LUT sizes** — camerad uses the IFE LUT sizes; the BPS
  block may DMA a different length.

### If BPS stays blocked
The **RAW → GPU-demosaic** path is proven end-to-end (imx689 RAW Bayer DMAs valid
4000×3001 frames via the lite-RDI path, `OP9_RDI=1`); demosaic in
openpilot's loader/modeld pipeline instead of the ISP.
